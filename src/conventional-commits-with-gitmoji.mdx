# Instructions to write commit messages

## Purpose

<purpose>
    You're an expert software engineer.
    Your purpose is to create consistent and meaningful commit messages using
    the [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/)
    and [Gitmoji](https://gitmoji.dev/) standards.
    You ensure a clear and understandable project history,
    facilitate easier collaboration, and enable automated tools to generate
    changelogs and manage versioning effectively.
</purpose>

## Decision Flow

<decision-flow>
Think step by step to generate 10 commit messages.
For each commit message:

    1. Analyze the changes introduced by the commit.
    2. Identify possible breaking changes.
    3. Generate the commit message subject.
        1. Choose the commit type.
        2. Identify the scope of the commit.
        2. Choose the Gitmoji according to the the intention of the commit.
        3. Write a short description to describe the changes.
    6. Add a body with a longer description of the changes if the subject doesn't
       describe the changes clearly.
    7. Add a footer if needed.
    8. Validate the commit message.

Choose the commit message that best describes the changes and passes validation.

</decision-flow>

## Commit Message Format

Generate commit messages following the
[Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) standard.
This ensures consistency in the commit history and enables automated changelog generation.

<instructions>
    <instruction>The first line of the commit message is the `{subject}` line.</instruction>
    <instruction>
    A longer commit `{body}` **may** be provided after the `{subject}`'s short `{description}`,
    providing additional contextual information about the code changes.
    </instruction>
    <instruction>
        The commit message **must** adhere to the following structure:

        <output-structure>

        ```md
        {subject}

        {body optional}

        {footer optional}
        ```

        </output-structure>
    </instruction>
</instructions>

### subject line

<instructions context="{subject}">
The `{subject}` has the following requirements:

    <rules>
    - **Structure**: <output-structure>`{type}{description}`</output-structure>
    - **Length**: Must be less than 100 characters
    - **Punctuation**: Must not end with punctuation
    - **Format**: Must not contain line breaks
    - **Content**: Must describe the commit's intent
    - **Description**: Must immediately follow the colon and space after the type/scope prefix
    </rules>

</instructions>

#### Commit Types

<instructions context="{type}">
    Choose the commit type noun from this list to categorize changes:

    - `feat`: A new feature
    - `fix`: A bug fix
    - `docs`: Documentation only changes (README file, license, `doc` folder contents)
    - `style`: Changes that don't affect the meaning of the code (formatting, removal of redundant information)
    - `refactor`: A code change that neither fixes a bug nor adds a feature
    - `perf`: A code change that improves performance
    - `test`: Adding missing tests or correcting existing tests (changes in `tests` folder)
    - `build`: Changes that affect the build system or external dependencies
      (`composer.json`, `Dockerfile`, package manager configuration, …)
    - `ci`: Changes to CI configuration files and scripts
      (linters, static code analytics, GitHub & GitLab CI pipelines, …)
    - `chore`: Other changes that don't modify the source code (`.gitignore`, …)
    - `revert`: Reverts a previous commit

    <rules>
    - Commits **must** be prefixed with a type, followed by the **optional** scope,
      **optional** `!` for breaking changes, and **required** terminal colon and space.
    - Type structure: `{type-noun}({scope optional}){breaking-change-flag optional}: `
    - **Always** use the type `feat` when a commit adds a new feature.
    - **Always** use the type `fix` when a commit represents a bug fix.
    </rules>

    <example>`feat(user-auth)!: `</example>
</instructions>

##### Commit Scopes

<instructions context="{scope}">
- A `{scope}` **may** be provided after a type, and is optional.
- A `{scope}` **must** consist of a noun describing a section of the codebase
    surrounded by parenthesis `()`.

    <examples>
    Example scopes:
    - `(auth)`
    - `(parser)`
    - `(api)`
    </examples>

- The `{scope}` should be the name of the module, package, component, or area affected
    by the changes (as perceived by the person reading the changelog).
- For changes affecting multiple modules or the entire codebase, either:
    - Omit the scope: `feat: ✨ add new feature`
    - Use a relevant high-level scope: `feat(core): ✨ improve performance`
- Choose a scope that provides meaningful context about where the change was made.
- Use consistent scope names across commits to make the changelog more organized.

</instructions>

#### Description

<instructions context="description">

- The `{description}` **must** begin with a `{gitmoji}`, followed by a space, and then a concise `{description-text}`.
- Format: `{gitmoji} {description-text}`
- The `{description-text}` **must**:
    - Use imperative, present tense (e.g., "add", "update", "fix", not "added", "updated", "fixed")
    - Be clear, specific, and descriptive without being verbose
    - Start with a lowercase letter
    - Not end with a period or other punctuation
    - Provide enough context to understand the change without reading the diff
    <examples>
    Example descriptions:
    - `✨ add user authentication system`
    - `🐛 fix memory leak in data processing loop`
    - `♻️ refactor user validation logic`
    - `📝 update API documentation`
    </examples>

</instructions>

### Body

<instructions context="{body}">
The `{body}` has the following requirements:

- Must begin one blank line after the `{subject}`.
- Must be wrapped at 72 characters.
- Use the body to provide additional context and details about the changes.
- Explain what and why compared to how.
- Is free-form and may consist of any number of newline separated paragraphs.
- Uses the Markdown format.
</instructions>

### Footer

<instructions context="{footer}">
The `{footer}` has the following requirements:

- One or more footers **may** be provided one blank line after the `{body}`.
- Each footer **must** consist of a word `{token}`, followed by either a `:<space>` or `<space>#` separator,
    followed by a string value (see the [git trailer convention](https://git-scm.com/docs/git-interpret-trailers)).
- The footer structures are:
    - <output-structure>`{token} #{issue_number} {text}`</output-structure>
    - <output-structure>`{token}: {text}`</output-structure>
- A footer's token **must** use `-` in place of whitespace characters,
    <example>for example: `Acked-by`</example> (this helps differentiate the footer section from a multi-paragraph body).
- An exception is made for `BREAKING CHANGE`, which **may** also be used as a token.
- A footer's value **may** contain spaces and newlines.
- Common footers include:
    - Issue references:
        - `Closes #{issue_number}`
        - `Fix #{issue_number}`
        - `Refs: #{issue_number}`
    - Sign-off: `Signed-off-by: {name} <{email}>`
    - Acknowledgements:
        - `Acked-by: {name} <{email}>`
        - `Helped-by: {name} <{email}>`
    - References to other commits:
        - `Reference-to: {commit_hash} ({commit_subject}, {commit_date})`
        - `See-also: {commit_hash} ({commit_subject})`
    - Breaking changes: `BREAKING CHANGE: {breaking_change_description}`
</instructions>

<example>

This is an example of a footer:

    ```text
    BREAKING CHANGE: environment variables now take precedence over config files.
    Closes #123
    Fix #123
    Refs: #34, #45
    Signed-off-by: Bob <bob@example.com>
    Acked-by: Alice <alice@example.com>
    Helped-by: Junio C Hamano <gitster@pobox.com>
    Reference-to: 8bc9a0c769 (Add copyright notices., 2005-04-07)
    See-also: fe3187489d69c4 (subject of related commit)
    ```

</example>

### Breaking Changes
### Breaking Changes

<instructions context="breaking-change">
Breaking changes **must** be indicated in one of two ways:

1. In the commit type prefix with a `!` character:
    - Add `!` immediately before the colon: `feat(auth)!: 💥 change authentication flow`
    - When using `!`, the commit description **must** use the `💥` Gitmoji
    - The commit body **must** contain details about the breaking change
2. As a footer entry:
    - Add `BREAKING CHANGE: {breaking-change-description}` to the footer section
    - The footer must start with the exact text `BREAKING CHANGE:` followed by a space
    - Example:

      ```md
      BREAKING CHANGE: environment variables now take precedence over config files
      ```

**Important rules:**

- A breaking change **must** be described in detail in the `{body}` of the commit message
- If the `!` notation is used in the type, the `BREAKING CHANGE:` footer **may** be omitted
- The commit message description for breaking changes **must** use the `💥` Gitmoji

</instructions>

## Conventions

<instructions>

- Files in `.github` are either related to Continuous Integration (CI) or
  contain instructions for GitHub Copilot (developer experience).
- Use `style` type and `:fire:` Gitmoji for removing redundant docblocks.
- Changes to `composer.json` **must** use the `build` type.
- **Do not** mention changes to dates in the commit message.

</instructions>

## Validation rules

<validation-rules>
    <rule context="subject">
        <constraint>maxLength: 100</constraint>
        <constraint>noPunctuation: true</constraint>
        <constraint>format: {type}({scope})!: {gitmoji} {description}</constraint>
    </rule>
    <rule context="body">
        <constraint>wrapAt: 72</constraint>
        <constraint>blankLineAfterSubject: true</constraint>
        <constraint>format: markdown</constraint>
    </rule>
</validation-rules>
